# 笔记格式规范

本文用于统一本仓库学习笔记、说明文档和阅读记录的写法。

如果是 `content/technology-introduction/` 下那种更正式的“技术主题页”，请同时参考 [TECH_TOPIC_STYLE.md](./TECH_TOPIC_STYLE.md)。

以后如果需要整理或重写文档，可以直接说明：

- 按 `docs/NOTE_STYLE.md` 优化
- 按仓库现有笔记风格整理
- 按当前笔记格式重写

默认都表示按本文档的规则执行。

## 1. 目标

本仓库文档主要面向 STM32 学习、项目回顾和后续复习，所以优先考虑下面几件事：

- 结构清晰，能快速找到重点
- 说明紧凑，不要一行只有几个字
- 内容实用，方便以后回头查
- 尽量减少重复，避免同一件事反复说
- 既能做学习笔记，也能作为项目说明使用

文档不是为了写得花，而是为了以后再看时能快速恢复理解。

## 2. 总体原则

### 2.1 先总览，再展开

一篇笔记通常按这个顺序组织：

1. 先说明本文要解决什么问题
2. 再给整体理解或总结
3. 然后用表格、流程图、分节说明展开
4. 最后补充当前项目中的对应关系、注意点或后续阅读建议

不要一开始就陷进局部细节，也不要上来就堆很多寄存器位定义。

### 2.2 说明要紧凑

优先使用短段落、紧凑表格和少量平铺列表。避免下面这种写法：

- 一个标题
- 下一行只有一句很短的话
- 再空一行
- 再来一句更短的话

目标是让文档读起来像正常文章，而不是被拆碎的提纲。

### 2.3 少重复，多归纳

如果前面的总览、流程图、表格已经把事情讲清楚，后面就不要再把同样内容重写一遍。后文应该承担的是：

- 补充说明
- 对比差异
- 结合当前项目举例
- 总结记忆点

而不是把前面的内容换个说法再写一次。

### 2.4 优先解释“作用”和“关系”

写代码、配置、寄存器或文件时，优先回答下面几个问题：

- 这个东西是做什么的
- 它和其他模块是什么关系
- 它在当前项目里怎么使用
- 学习时应该重点关注什么

不要一上来就堆参数、宏定义或寄存器位。

## 3. 标题与层级

### 3.1 使用编号标题

正式笔记默认使用编号标题，例如：

```markdown
## 1. GPIO 是什么
## 2. GPIO 能做什么
## 3. 常见模式
## 4. 结合项目理解
```

子章节继续使用：

```markdown
### 2.1 结构图里的关键模块
### 2.2 当前项目对应关系
```

### 3.2 标题层级不要太深

建议最多使用到 `###`，只有确实需要时才使用 `####`。不要为了显得详细而层层嵌套。

### 3.3 标题要有信息量

标题尽量直接表达该节内容，避免使用太空泛的名字，例如：

- 推荐：`GPIO 分类、常见模式与记忆重点`
- 不推荐：`更多说明`

## 4. 段落、列表和表格

### 4.1 段落以“短而完整”为准

一个段落通常表达一个完整意思，长度以读起来顺为准。一般 `2` 到 `5` 行比较合适，太短会显得碎，太长会显得闷。

### 4.2 列表只在确实需要并列时使用

适合用列表的场景：

- 目标
- 特点
- 注意事项
- 对比项
- 阅读顺序

如果只是正常解释概念，优先写成连续短段落，不要强行列表化。

### 4.3 避免多层嵌套列表

尽量使用平铺列表。若某一项需要补充说明，优先在该项下面接一小段话，而不是继续缩进很多层。

### 4.4 表格优先用于快速归纳

出现下面这些内容时，优先考虑表格：

- 多个文件或目录的作用
- 多种 GPIO 模式的区别
- 内部模块及其功能
- 引脚分配
- 时钟参数汇总

表格适合做“快速查阅”，正文负责做“理解说明”。两者配合使用，不互相替代。

### 4.5 表格列名尽量稳定

本仓库里可优先使用这些常见表头：

- `模块 | 基本作用 | 直观理解 | 说明`
- `模式 | 类别 | 基本含义 | 原理或关键特点 | 常见用途`
- `文件或目录 | 主要作用`
- `引脚 | 功能 | 当前用途 | 说明`

## 5. 内容编排建议

### 5.1 概念类笔记

例如 GPIO、时钟、串口这类基础知识，建议使用下面结构：

1. 先解释“它是什么”
2. 再解释“它能做什么”
3. 再给结构图、模块图或模式总表
4. 最后给记忆重点和项目对应关系

如果这篇内容后续会沉淀成长期可复用的“章节页”，建议把正式版放到专题目录下，再保留一份较完整的学习笔记。

### 5.2 文件阅读类笔记

例如项目文件、启动流程、HAL 与寄存器关系，建议使用下面结构：

1. 阅读目标
2. 初步理解总结
3. 基本流程图或关系图
4. 快速查表
5. 必要补充说明
6. 建议阅读顺序

### 5.3 项目说明类文档

例如项目结构、引脚占用、时钟配置，建议使用下面结构：

1. 文档目的
2. 当前状态或总览
3. 参数汇总表
4. 分项说明
5. 当前项目注意事项

## 6. 图和流程图

### 6.1 流程图服务于“整体理解”

如果内容涉及启动流程、工程生成流程、模块关系，可以使用 `mermaid`。流程图重点是帮助建立整体印象，而不是把每个函数都拆开。

要求如下：

- 节点数量适中，不要太碎
- 优先体现主干流程和关键依赖
- 一个框里可以写两行，但不要塞太多句子
- 文件名和作用可以合并在同一个框里展示

### 6.2 图片前后要有解释

插入结构图或示意图后，前后要补一句说明：

- 这张图是为了看什么
- 读图时应该先关注哪几个模块

不要只贴图不解释。

## 7. 代码、配置和参数说明

### 7.1 先说作用，再说参数

遇到代码或配置项时，优先这样写：

1. 这个配置是做什么的
2. 为什么这样配
3. 当前项目里是怎么配的
4. 需要注意什么

### 7.2 参数尽量成组整理

例如时钟配置，不要把同一组参数散在多个段落里，优先集中列出：

- `HSE = 8MHz`
- `PLL = x9`
- `SYSCLK = 72MHz`
- `APB1 = 36MHz`
- `APB2 = 72MHz`

### 7.3 示例代码保持最小必要

代码块只保留理解必须的部分，不贴整份源码。目标是帮助理解，不是复制整文件。

## 8. 与当前项目结合

本仓库笔记建议尽量和当前 STM32 项目建立联系，而不是只写抽象定义。常见写法包括：

- “放到当前项目里，可以这样理解……”
- “当前工程中对应的是……”
- “在本项目里，这个引脚用于……”

这样后续复习时更容易把知识点和代码对应起来。

## 8.1 主题页与笔记分工

当同一个主题同时存在“正式主题页”和“学习笔记”时，建议这样分工：

- 正式主题页：总览、配图、核心表格、常见错误、项目对应关系
- 学习笔记：阅读过程、细节展开、临时理解、后续问题

这样可以避免一份文档既想做手册，又想保留全部思考过程，最后变得过重。

## 9. 路径和链接规范

### 9.1 一律使用相对路径

仓库内 Markdown 链接统一使用相对路径，不使用绝对路径，例如：

- 正确：`[README.md](../README.md)`
- 正确：`[main.c](../../Core/Src/main.c)`
- 错误：`D:/code/...`

### 9.2 路径以当前文档位置为基准

例如：

- 根目录 `README.md` 引用 `docs/README.md`，写成 `./docs/README.md`
- `docs/README.md` 引用根目录 `README.md`，写成 `../README.md`
- `content/technology-introduction/mcu-basics/01-project-files.md` 引用 `CMakeLists.txt`，写成 `../../../CMakeLists.txt`

## 10. 文风要求

整体风格保持：

- 清晰
- 平实
- 紧凑
- 不堆术语
- 不故意写得很学术

推荐句式：

- “这个文件负责什么”
- “这里的重点是”
- “可以先这样理解”
- “放到当前项目里”
- “学习时先抓住这几点”

避免：

- 太多空话
- 太多口号式表达
- 明明很简单却写得很绕

## 11. 优化已有文档时的默认动作

如果用户要求“按现有风格优化”或“按 `docs/NOTE_STYLE.md` 调整”，默认执行下面这些事：

1. 保留原有有效信息
2. 删除重复表达
3. 合并过碎的小节
4. 补齐编号标题
5. 需要时增加表格或流程图
6. 把说明改得更紧凑
7. 将路径统一成相对路径
8. 增加与当前项目的对应说明

## 12. 一句话触发规则

以后如果出现下面这些说法，都默认按本规范执行：

- 按 `docs/NOTE_STYLE.md` 优化
- 按仓库现有笔记风格整理
- 按这种笔记格式重写
- 调整成和当前文档一致的风格

如果没有特别说明，默认目标是：保留内容有效性，优化结构和表达，让文档更适合长期学习和快速复习。
