# 技术主题页格式规范

本文用于统一 `content/technology-introduction/` 下“正式主题页”的写法，例如 `GPIO`、`PWM`、`I2C`、`SPI`、`ADC` 这类可长期复用的基础章节。

它和 [NOTE_STYLE.md](./NOTE_STYLE.md) 的关系是：

- `NOTE_STYLE.md` 偏向通用笔记、阅读记录、项目说明
- 本文偏向正式主题页，强调“像手册章节一样可持续扩展”

## 1. 适用范围

适用于下面这类文档：

- 某项基础能力介绍，例如 `GPIO`、`PWM`
- 某类接口介绍，例如 `I2C`、`SPI`、`UART`
- 某类通用原理介绍，例如中断、时钟、ADC、滤波

不适用于下面这类文档：

- 单次阅读记录
- 临时项目笔记
- 具体芯片某一页寄存器摘录

## 2. 主题页目标

主题页不是把资料原文搬过来，而是做成便于长期查看的“总览章节”。默认应同时满足：

- 初看时能快速建立整体认识
- 回看时能快速定位关键表格和配图
- 后续可继续追加更多子主题、案例图和项目关联

## 3. 推荐目录结构

每个主题建议单独建目录：

```text
content/technology-introduction/<category>/<topic-id>/
├─ README.md
├─ images/
│  ├─ overview/
│  ├─ structure/
│  ├─ timing/
│  └─ examples/
└─ references/
```

说明：

- 正文固定放在 `README.md`
- 图片按用途分组，而不是混放
- 如果当前主题图片较少，可以只保留 `images/`
- `references/` 可选，用于保存资料截图、整理图或额外图示

## 4. 推荐章节结构

正式主题页默认按下面顺序组织：

1. `这是什么`
2. `结构图 / 原理图`
3. `关键组成或关键模式`
4. `工作过程 / 时序过程`
5. `常见用途`
6. `常见错误`
7. `结合当前项目理解`
8. `关联内容`

不是每一页都必须完整照搬，但整体顺序尽量保持稳定。

## 5. 配图要求

### 5.1 图片必须服务于阅读

每张图都应该解决一个明确问题，例如：

- 这项技术内部由哪些模块组成
- 信号从哪里进、从哪里出
- 时序是怎样变化的
- 当前项目中是如何接线或使用的

不要只因为“有图”就加入图片。

### 5.2 图片前后必须有说明

插图前后至少补一句：

- 这张图主要看什么
- 读图时先关注哪几个模块或路径

### 5.3 图片路径规范

统一使用相对路径，优先写成：

```markdown
![GPIO 基本结构示意](../images/02-gpio-basic.png)
```

或主题独立目录内：

```markdown
![PWM 时序示意](./images/timing/pwm-basic.svg)
```

### 5.4 SVG 图优先适配在线查阅

技术主题图不仅要在本地 Markdown 中可读，也要考虑后续 GitHub Pages 在线查看。

要求：

- 默认按桌面浏览器宽度优先排版
- 标题、主示意区、说明区要明显分层
- 不把长说明直接压在波形或结构图上
- 关键说明尽量控制在短句，避免一整段横向铺满
- 纯解释性总结优先放在底部说明栏

## 6. 内容密度要求

主题页应偏“简洁总览”，不要写成纯资料摘录。默认建议：

- 一张总览图
- 一张关键表格
- 一张工作流程或时序图
- 少量结论性解释

如果某部分需要展开，优先放到配套笔记里，再从主题页链接过去。

## 7. 表格规范

主题页尽量多用表格压缩空间。优先使用这些表头：

- `模块 | 基本作用 | 直观理解 | 说明`
- `模式 | 类别 | 基本含义 | 关键特点 | 常见用途`
- `关注点 | 读图时看哪里 | 先理解什么`
- `引脚/信号 | 当前用途 | 对应理解 | 说明`
- `类型 | 内容 | 说明`

要求：

- 列名尽量稳定
- 表格优先承担“查阅”职责
- 正文只补充最必要的解释

## 8. 主题页与笔记的关系

同一主题可以同时存在两份文档：

- 主题页：更像手册章节，面向长期查看
- 笔记页：更像学习记录，允许保留探索过程和展开解释

推荐做法：

- `content/technology-introduction/mcu-basics/gpio/README.md`
  作为正式主题页
- `content/technology-introduction/mcu-basics/02-gpio-basics.md`
  作为配套学习笔记

二者应互相链接，但不要简单复制。

## 9. 文风要求

整体风格保持：

- 简洁
- 可查
- 可复用
- 有图可看
- 有表可比

避免：

- 大段资料原文搬运
- 只有文字没有图表
- 章节顺序随意变化太大
- 同一个概念反复重复

## 10. 默认执行规则

以后如果用户说：

- 按主题页风格整理
- 做成技术介绍页
- 按手册章节方式整理

默认都按本文执行，并同时参考 [NOTE_STYLE.md](./NOTE_STYLE.md)。

## 11. GitHub Pages 适配建议

如果后续发布到 GitHub Pages，主题页建议优先满足：

- 目录清晰，能从首页逐级进入主题页
- 图片全部使用相对路径
- 页面即使没有复杂前端，也能直接在线阅读
- 未来如果切换到静态站点生成器，当前 Markdown 和图片结构仍能继续复用

当前更稳妥的路线是：

1. 先把仓库内容按 Markdown 手册方式整理好
2. 保持 `content/`、`docs/`、`images/` 结构稳定
3. 后续再决定是直接用 GitHub Pages 发布静态文件，还是接入 Jekyll / MkDocs / VitePress 一类生成器
